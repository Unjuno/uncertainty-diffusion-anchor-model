# Concrete Example: Work Re-Anchor

This example shows how UDAM can be used when a work project feels unclear.

## Situation

A person has avoided a project for a while.

They do not know the current project state.

## Lost anchor

The project state is no longer calibrated.

Possible states:

```text
clear
blocked
outdated
waiting
```

## Common mistake

The person treats unclear project state as a reason to avoid opening the project.

```text
I do not know where this stands -> I avoid looking
```

## Re-anchor action

Open the task list or working document.

Spend ten minutes identifying only the next concrete blocker.

Do not try to finish the project during this step.

## If you are in this situation, do this now

```text
1. Open the task list or working document.
2. Spend 10 minutes only.
3. Write the next blocker or the next clear action.
```

Do not try to finish the project in this pass.

The goal is to reveal the current project state.

## Observation result

The result separates possible states:

```text
next step is clear
one item is blocked
document is outdated
someone needs a reply
```

Each result suggests a different next action.

## Next action map

```text
next step is clear -> do the first small task
one item is blocked -> name the blocker
document is outdated -> update the first outdated section
someone needs a reply -> send one specific reply
```

## UDAM interpretation

The action has observability value because it converts vague project uncertainty into a local decision state.

```text
OV > 0
```

If the project state has drifted, early observation can still reduce future cost.

## Best practice

```text
open project -> identify next blocker -> choose one next action
```

## Boundary

Do not keep checking the task list if the next action is already clear.

Repeated checking is useful only while:

```text
MOV_i > 0
```
