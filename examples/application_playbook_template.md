# Application Playbook Template

Use this template when adding concrete UDAM applications.

The goal is practical clarity.

Do not make the protocol complicated.

Every application should answer:

```text
What should the reader do now?
```

## 1. Situation

Describe the concrete situation.

Keep it recognizable.

```text
The person is avoiding X because the state of X is unclear.
```

## 2. Lost anchor

Name what is no longer calibrated.

```text
skill state
project state
social state
budget state
body-state trend
option state
```

## 3. Possible states

List 3 or 4 possible states.

```text
state 1
state 2
state 3
state 4
```

The goal is not perfect taxonomy.

The goal is to stop treating uncertainty as one global mass.

## 4. Common mistake

Write the mistaken inference.

```text
unclear state -> global conclusion -> avoidance or wrong action
```

## 5. Re-anchor action

Give one small state-informative observation.

It must be connected to the state.

```text
P(y | S) != P(y)
```

Practical test:

```text
What result would change the next action?
```

## 6. If you are in this situation, do this now

Give 3 concrete steps.

```text
1. Do one small observation.
2. Limit it by time, count, or scope.
3. Classify the result.
```

This section is required for practical examples.

## 7. Observation result

List possible results.

```text
result 1
result 2
result 3
result 4
```

## 8. Next action map

Map each result to a next action.

```text
result 1 -> next action 1
result 2 -> next action 2
result 3 -> next action 3
result 4 -> next action 4
```

This is the most important section.

UDAM becomes practical only when observation changes action.

```text
a(y) != a_0
```

## 9. UDAM interpretation

Explain why the action has observability value.

```text
OV > 0
```

Use plain language:

```text
The action reveals the current state and allows a better next action.
```

## 10. Boundary

State when to stop checking.

```text
If another observation would not change the next action, stop checking and act.
```

Repeated observation is useful only while:

```text
MOV_i > 0
```

## Minimal playbook shape

```text
unclear state
-> state-informative observation
-> classify result
-> choose mapped next action
```

## Bad application shape

Avoid this:

```text
unclear state -> do something vague -> feel better
```

UDAM is not vague motion.

UDAM is state-informative re-anchoring.
