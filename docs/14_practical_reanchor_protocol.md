# 14 Practical Re-Anchor Protocol

This document answers the practical question:

```text
I understand UDAM. What should I do now?
```

UDAM should not remain only an abstract model.

It should become a usable protocol for unclear states.

## Simple version

Use this first.

```text
1. Name what is unclear.
2. Take one small observation connected to that state.
3. Classify the result.
4. Do the next action that matches the result.
```

The whole protocol can be summarized as:

```text
unclear state -> observe one useful signal -> map result to next action
```

Do not make the first step large.

The first step is not the whole solution.

It is a state check.

## The 7-step protocol

Use this when a situation feels unclear and you are tempted to freeze, avoid, or guess globally.

## Step 1: Name the domain

Write one phrase for the domain.

Examples:

```text
study
work
relationship
budget
health tracking
life planning
```

Do not solve yet.

Only name the domain.

## Step 2: Name the hidden state

Ask:

```text
What state am I actually uncertain about?
```

Examples:

```text
skill state
project state
social state
budget state
body-state trend
option state
```

This converts vague pressure into a latent state `S_t`.

## Step 3: List 3 or 4 possible states

Write a small state set.

Example for study:

```text
retained
partial
missing
```

Example for work:

```text
clear
blocked
outdated
waiting
```

Example for social uncertainty:

```text
busy
neutral
unclear
less engaged
```

The goal is not perfect classification.

The goal is to stop treating uncertainty as one global mass.

## Step 4: Choose one small state-informative observation

Ask:

```text
What is the smallest action that would tell me which state I am closer to?
```

A useful observation must be connected to the state.

Formal test:

```text
P(y | S) != P(y)
```

Practical test:

```text
What result would change my next action?
```

Good observations are:

- small;
- concrete;
- low cost;
- tied to the real domain;
- able to change the next action.

Examples:

```text
solve one representative problem
open the task list
send one low-pressure message
write down three planning numbers
```

## Step 5: Time-box it

Set a small limit.

Examples:

```text
5 minutes
10 minutes
one message
one page
one problem
three numbers
```

The first re-anchor is not the whole solution.

It is a measurement.

## Step 6: Map observation results to next actions

Before acting, define what different results would mean.

Example:

```text
if I cannot start -> review the first concept
if I start but get stuck -> find the stuck step
if I solve with notes -> repeat without notes
if I solve without notes -> move to the next topic
```

This is the key point.

Observation is valuable because it enables conditional action.

```text
observe y -> choose a(y)
```

A strong re-anchor usually satisfies:

```text
a(y) != a_0
```

for at least one possible observation result.

## Step 7: Stop or continue by marginal value

After one observation, ask:

```text
Would another observation change my next action?
```

If yes, another observation may be useful.

If no, stop checking and act.

Repeated observation is justified only when:

```text
MOV_i > 0
```

or in simpler form:

```text
I_i + B_i > C_i
```

## Decision card

Use this short card in daily life.

```text
1. What domain is unclear?
2. What hidden state am I uncertain about?
3. What are 3 possible states?
4. What small observation is connected to that state?
5. What is the time box?
6. What result would change my next action?
7. After observing, do I act or observe once more?
```

## Four daily templates

### Study

```text
unclear skill -> one representative problem -> choose next study action
```

### Work

```text
unclear project -> open task list -> identify next blocker
```

### Relationship

```text
unclear social state -> one low-pressure message -> update from response pattern
```

### Budget

```text
unclear budget -> write three planning numbers -> choose next adjustment
```

## What not to do

Do not ask first:

```text
Can I solve the whole problem?
```

Ask:

```text
What small observation would reveal the current state?
```

Do not keep observing after the next action is already clear.

Do not use UDAM to justify vague activity.

UDAM supports informative action, not motion for its own sake.

## Core practical rule

```text
unclear state -> state-informative observation -> conditional next action
```

If the observation cannot change belief, decision, or state, it is probably not a re-anchor.
