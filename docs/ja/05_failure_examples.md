# 失敗例

この文書は、UDAM の誤用を見分けるための具体例です。

UDAM は「とにかく小さく行動すればいい」という話ではありません。

有効なのは、状態について情報を返し、次の行動を変えられる観測です。

## 1. 状態と関係ない観測

### 状況

勉強できる状態か知りたい。

本当に知りたい状態は：

```text
S = 今、問題を解けるか
```

しかし、実際には問題を解かずに、ノート整理やファイル整理だけをしている。

### なぜ失敗か

ノートが整理されたことは、問題が解けることを直接示しません。

```text
ノートが整理された ≠ 問題が解ける
```

形式的には：

```text
P(y | S) = P(y)
```

つまり、観測 `y` が状態 `S` と関係していない。

### 修正

状態と関係する観測にする。

```text
代表的な問題を1問解く
```

その結果で次を決める。

```text
解けた -> 少し広げる
途中まで解けた -> 抜けている手順を見る
解けない -> 前提知識に戻る
```

## 2. 確認ループ

### 状況

作業の状態がわからない。

作業リストを1回見た。

その結果、次にやるべきタスクはわかった。

しかし、行動せずに何度も同じリストを見返している。

### なぜ失敗か

最初の確認は有効でした。

しかし、2回目以降の確認で次の行動が変わらないなら、価値は下がります。

```text
同じ情報 -> 同じ行動 -> 追加コスト
```

条件：

```text
MOV_i <= 0
```

### 修正

問いを変える。

```text
次の確認で、次の行動が変わるか？
```

変わらないなら、確認ではなく行動する。

```text
1つ目のタスクに着手する
```

## 3. 広げすぎ

### 状況

10分作業して、状態が少し見えた。

それがうまくいったので、次にいきなり4時間作業しようとする。

```text
10分 -> 4時間
```

### なぜ失敗か

小さい範囲でうまくいったことは、大きい範囲でも安全という証明ではありません。

```text
前の範囲で大丈夫だった = 次の範囲でも大丈夫
```

ではない。

正しくは：

```text
前の範囲で大丈夫だったことは、次の範囲についての証拠の一部
```

拡張が失敗する条件：

```text
B_expand(r_i) + I_expand(r_i)
<=
C_obs(r_i) + P_boundary(i) * C_boundary + C_correct(r_i)
```

### 修正

広げるとしても、段階を小さくする。

```text
10分 -> 20分 -> 40分
```

または、途中で観測点を入れる。

```text
30分やる -> 状態を確認する
```

## 4. 局所観測から全体結論を出す

### 状況

1問解けなかった。

そこから、こう結論してしまう。

```text
自分はこれを学べない
```

### なぜ失敗か

1問解けなかったことは、意味のある観測です。

しかし、それは局所観測です。

更新できるのは：

```text
この問題タイプが今は難しい
```

という局所的な信念です。

そこからすぐに：

```text
自分は全部だめ
```

とは言えません。

失敗パターン：

```text
局所結果 -> 全体断定
```

### 修正

観測の範囲を守る。

```text
この1問はできなかった
```

次に、より小さい前提を観測する。

```text
前提問題を1つ解く
似た簡単な問題を1つ解く
解説のどこで止まったかを見る
```

## まとめ

UDAM の主な誤用は次の4つです。

```text
状態と関係ない観測
確認ループ
広げすぎ
局所観測から全体断定
```

短く言うと：

```text
小さく見るだけでは足りない。
何を見るか、見たあと何を変えるか、どこまで広げるかが重要。
```
